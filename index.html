A change to overflow on replaced elements in CSS

From Chrome 108, the following replaced elements respect the overflow property: img, video and canvas. In earlier versions of Chrome, this property was ignored on these elements. This means that an image which was earlier clipped to its content box can now draw outside those bounds if specified to do so in a style sheet. The default browser style sheet clips the overflow to the content box with the following rules: img { overflow: clip; overflow-clip-margin: content-box; } However, it is possible for your stylesheet to override this behavior by setting overflow: visible. The following examples show some cases where you might now see unwanted overflow. The object-fit property is used to scale the image and fill the box. If the aspect ratio does not match the box, the image will draw outside of the bounds. img { object-fit: cover; overflow: visible; } The border-radius property makes a square image look like a circle, but because overflow is visble the clipping no longer occurs. img { border-radius: 50% 50%; overflow: visible; } Setting inherit: all and causing all properties to inherit, including overflow. img { all: inherit; } # Solution If this change means you see unwanted overflow, and overriding the overflow property to visible is not intentional, inspect the CSS applied to the element via DevTools. Many apps don't fail stable in the sense that if there is a server error they don't present the last known good view of data. What this means in practice is if there is low network availability (on a train, at a concert or festival, a sporting event) many of these apps completely break. Back years ago I would design apps to sync to a local database (which was relatively easy to do on IOS). Apps would maintain consistent state on the device and leverage local storage. I haven't looked under the hood but many apps appear to be using HTMLish front ends that are stateless, except for some cached login information. As a user - what I see is that the app repeatedly tries to login and fetch all information every time you load it. Which if you have perfect LTE/5G/WIFI is sort of ok, but in other situations sucks. if the app can't talk to the network it really doesn't work at all. In practice this means significant latency between seeing the message on the phone and then going to the conversation view (which can take forever to load under poor networking).
